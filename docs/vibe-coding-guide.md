# 바이브 코딩 가이드 — club-house 관리하기

Vibe Coding Club 홈페이지(club-house)를 동아리원 누구나 관리할 수 있도록 만든 가이드입니다.
개발 지식이 없어도 괜찮습니다. **AI 코딩 도구와 한국어로 대화하면서** 사이트를 수정하고 배포할 수 있습니다.

## 큰 그림

이 프로젝트는 아래처럼 굴러갑니다.

```
AI 도구로 수정 → GitHub에 push → Vercel이 자동으로 배포 (1~2분)
```

배포 버튼을 누르거나 서버를 만질 일이 없습니다. **GitHub에 코드가 올라가면 사이트가 자동으로 갱신됩니다.**

- 코드 저장소: https://github.com/vibe-ai-coding-club/club-house
- 배포: Vercel (GitHub 연동 자동 배포)

## 준비물 (최초 1회)

1. **GitHub 계정** — 없다면 https://github.com 에서 가입 후, 운영진에게 저장소 접근 권한(collaborator)을 요청하세요.
2. **AI 코딩 도구** — 아무거나 하나면 됩니다. 익숙한 것을 쓰세요.
   - **Claude Code** (맥 앱 / CLI) — 에디터 없이 대화만으로 작업 가능
   - **Cursor** — VS Code 스타일 에디터 + AI
   - **Codex CLI**, **Gemini CLI**, **Windsurf** 등 — 어떤 도구든 아래 루틴은 동일합니다

이 문서의 예시는 도구에 상관없이 통하는 자연어 요청 기준으로 작성했습니다.

## 최초 세팅

사용하는 AI 도구를 열고 아래처럼 순서대로 부탁하면 됩니다. 명령어를 외울 필요 없이, 그대로 말하면 AI가 알아서 해줍니다.

1. **GitHub 로그인**

   > "GitHub에 로그인하고 싶어. 도와줘"

   AI가 인증 과정(`gh auth login` 또는 도구 자체 연동)을 안내해 줍니다. 브라우저가 열리면 GitHub 계정으로 인증하세요.

2. **프로젝트 받아오기**

   > "https://github.com/vibe-ai-coding-club/club-house 를 클론해서 열어줘"

   Cursor처럼 에디터 기반 도구라면 클론 후 그 폴더를 열면 됩니다.

3. **실행해 보기**

   > "이 프로젝트를 로컬에서 실행해서 보여줘"

   AI가 의존성 설치(`npm install`)와 개발 서버 실행(`npm run dev`)을 해줍니다.
   브라우저에서 http://localhost:3000 을 열면 사이트가 보입니다.

여기까지 되면 세팅 끝입니다. 다음부터는 프로젝트 폴더를 열고 바로 작업하면 됩니다.

## 평소 작업 루틴

모든 작업은 AI와의 대화로 진행합니다. 예시:

1. **수정 요청** — 원하는 걸 한국어로 설명하세요.
   > "메인 페이지 히어로 문구를 '○○○'로 바꿔줘"
   > "멤버 페이지에 새 멤버를 추가해줘. 이름은 ○○, 역할은 디자이너야"
   > "공지사항에 8월 모임 안내를 추가해줘"

2. **확인** — 로컬에서 눈으로 확인하세요.
   > "개발 서버 띄워줘"

   http://localhost:3000 에서 수정된 화면을 확인합니다.

3. **배포** — 마음에 들면 올려달라고 하세요.
   > "커밋하고 푸시해줘"

   1~2분 뒤 실제 사이트에 반영됩니다.

### 커밋 메시지

AI에게 맡기면 알아서 작성해 줍니다. 이 프로젝트는 `fix:`, `chore:`, `update:` 같은 접두어를 쓰는 관례가 있는데, "이 프로젝트 커밋 관례에 맞춰줘"라고 한마디 덧붙이면 됩니다.

## 자주 수정하는 곳

무엇을 바꿔달라고 할지 감을 잡는 용도입니다. 직접 파일을 열 필요는 없고, AI에게 "○○ 바꿔줘"라고 하면 알아서 찾아갑니다.

| 내용 | 위치 |
|------|------|
| 멤버 목록 | `src/data/members.ts` |
| 공지사항 | `src/data/notices.ts` |
| 프로젝트 아카이브 | `src/data/projects.ts` |
| 블로그 링크 | `src/data/blogs.ts` |
| 2026년 활동 계획 | `src/data/2026-plan.json` |
| 사이트 기본 정보(제목, 링크 등) | `src/config/site.ts` |
| 상단 메뉴 | `src/config/navigation.ts` |
| 페이지 화면 | `src/app/` 아래 각 폴더의 `page.tsx` |
| 화면 구성 요소(버튼, 카드 등) | `src/components/` |
| 이미지 등 정적 파일 | `public/` |

## 조금 더 안전하게 배포하기 (선택)

바로 배포하기 불안하다면, **브랜치**로 올려서 미리보기로 확인할 수 있습니다.

1. > "지금 수정한 내용을 새 브랜치로 만들어서 푸시해줘"
2. Vercel이 그 브랜치 전용 **미리보기 URL**을 만들어 줍니다. (GitHub의 Pull Request 페이지나 커밋 체크에서 링크 확인)
3. 미리보기가 괜찮으면:
   > "이 브랜치를 main에 머지해줘"
4. main에 머지되는 순간 실제 사이트에 배포됩니다.

문구 수정 같은 가벼운 변경은 main에 바로 푸시해도 충분하고, 레이아웃을 크게 바꾸는 작업은 브랜치 + 미리보기를 권장합니다.

## 뭔가 잘못됐을 때

- **배포 후 사이트가 깨졌어요** — 당황하지 않아도 됩니다. Vercel 대시보드에서 이전 배포를 선택해 **Instant Rollback**으로 즉시 되돌릴 수 있습니다. 대시보드 접근이 안 되면 AI에게 "마지막 커밋을 되돌려서 푸시해줘"라고 해도 됩니다.
- **로컬에서 실행이 안 돼요** — 에러 메시지를 그대로 AI에게 보여주세요. "이 에러 고쳐줘"면 대부분 해결됩니다.
- **푸시가 안 돼요** — GitHub 권한 문제일 가능성이 높습니다. 저장소 collaborator로 추가됐는지 운영진에게 확인하세요.
- **내가 뭘 바꿨는지 모르겠어요** — > "지금까지 뭘 수정했는지 정리해서 알려줘"

## 운영진 참고 사항

- Vercel 프로젝트는 현재 개인 계정 스코프(`da-ins-projects`)에 연결되어 있습니다. 배포 대시보드(롤백, 도메인, 배포 로그)를 관리할 사람은 해당 Vercel 프로젝트에 멤버로 초대되거나 소유권을 이전받아야 합니다.
- 이 프로젝트는 별도의 환경변수나 데이터베이스가 없어서, 필요 시 새 Vercel 계정에서 GitHub 저장소를 다시 Import 하는 것만으로 배포를 이관할 수 있습니다.
- 새 멤버 온보딩은 이 문서 링크 하나로 충분하도록 유지해 주세요. 프로젝트 구조가 바뀌면 위 표를 함께 갱신해 주세요.
